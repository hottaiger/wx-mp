# 录入前隐私授权技术设计

## 1. 设计目标

本设计修复微信审核失败原因 2：用户在“记一笔”录入人、事、物信息时，系统必须先完整告知《用户服务协议》和《隐私政策》，并在用户主动同意后才能把信息发送到微信云开发环境。

OpenSpec delta spec 是行为事实源：

- `openspec/changes/require-privacy-consent-before-capture/specs/privacy-consent/spec.md`
- `openspec/changes/require-privacy-consent-before-capture/specs/quick-capture/spec.md`

## 2. 当前数据外传路径

录入页存在两个需要授权守卫的入口：

1. 物品图片入口：`onChooseItemImage` 选择图片后立即调用 `wx.cloud.uploadFile`，早于最终保存。
2. 保存入口：`onSubmit` 可能请求订阅消息授权，并通过业务云函数写入人、事、物和提醒数据。

只在 `onSubmit` 增加复选框校验会遗漏图片云存储上传，无法形成完整合规闭环。

## 3. 方案比较

### 方案 A：所有协议和授权逻辑直接写入录入页

优点是改动文件少。缺点是协议正文、版本、存储和页面交互耦合在已经较长的 `pages/capture/index.js` 中，难以独立测试，后续协议升级容易漏改。

### 方案 B：独立授权模块 + 录入页弹层（采用）

`utils/privacy-consent.js` 负责协议与授权领域逻辑，录入页只负责展示和调用统一守卫。该方案不新增路由，不破坏表单现场，模块边界清晰，协议升级和测试成本低。

### 方案 C：新增两个独立协议页面

页面语义最独立，但需要新增路由、页面文件和返回现场处理；对当前单一录入入口属于过度设计，并增加审核路径和维护面。

## 4. 模块设计

### 4.1 `miniprogram/utils/privacy-consent.js`

职责：

- 导出当前协议版本和本地存储 key。
- 提供结构化《用户服务协议》《隐私政策》正文。
- 读取并校验 `{ version, agreedAt }` 授权记录。
- 写入当前版本授权记录。
- 撤回授权并删除记录。

公开接口保持小而稳定：

```js
getAgreement(type)
isConsentValid(storage)
grantConsent(storage, now)
revokeConsent(storage)
```

`storage` 默认为微信同步存储接口，同时允许测试注入内存适配器。协议正文采用 `sections: [{ title, paragraphs }]`，避免在 WXML 中解析 HTML 或把大段文本散落到页面文件。

协议版本使用日期格式，例如 `2026-07-13`。读取授权时必须同时满足：记录为对象、版本完全一致、`agreedAt` 为有效正数。任何异常都按未授权处理。

### 4.2 录入页状态

新增状态：

- `privacyAgreed`：当前版本授权是否有效。
- `privacyDialogVisible`：协议弹层是否显示。
- `privacyDialogTitle`、`privacyDialogSections`：当前弹层内容。

`onLoad` 从授权模块恢复状态。协议版本变化后旧记录自动失效，页面恢复为未勾选。

### 4.3 统一授权守卫

页面新增 `ensurePrivacyConsent()`：

1. 同时检查页面 `privacyAgreed` 和 `privacyConsent.isConsentValid()`。
2. 任一检查失败时同步恢复未勾选状态。
3. 显示固定提示“请先阅读并同意用户服务协议和隐私政策”。
4. 返回布尔值供调用方立即退出。

`onChooseItemImage` 和 `onSubmit` 必须在任何数据选择、上传、loading、订阅授权或云函数调用前调用该守卫。这样新增数据外传入口时，代码审查可以沿用同一约束。

## 5. 用户交互

保存 dock 上方展示协议确认区，复选框默认不预选。用户点击协议名称打开可滚动弹层；关闭弹层不会清空表单。

主动勾选时写入当前协议版本与同意时间。取消勾选时立即删除授权记录。授权只表示允许按当前政策处理用户主动输入的数据，不改变现有 `_openid` 数据隔离规则。

弹层正文必须覆盖：

- 数据类型：姓名、事项、物品、标签、备注、时间、属性、可选图片、提醒信息。
- 使用目的：记录管理、实体关联、周复盘和用户主动开启的提醒。
- 处理方式：通过微信云开发云函数或云存储传输，在云数据库/云存储中保存。
- 使用范围：仅向当前微信用户提供功能，不出售，不用于广告营销。
- 用户控制：可以删除记录、取消授权；取消后停止新增数据上传，已保存记录仍可按现有删除功能处理。

## 6. 错误与边界处理

- 微信本地存储读取异常：视为未授权，不阻断页面加载。
- 写入授权记录失败：保持未授权并提示用户稍后重试。
- 撤回存储删除失败：页面仍恢复未勾选，后续守卫重新校验持久化状态；测试覆盖失败降级。
- 用户未授权点击图片：不调用图片选择器和云存储上传。
- 用户未授权点击保存：不执行表单校验后的任何业务副作用。
- 提交过程中取消授权：提交按钮已处于 `submitting` 状态；当前提交按点击时的有效授权执行。下一次操作重新校验。

## 7. 测试策略

### 7.1 授权模块单元测试

使用 `node:test` 和内存 storage stub 覆盖：

- 无记录、损坏记录、旧版本记录均无效。
- 当前版本记录有效。
- 同意写入版本和时间。
- 撤回删除记录。
- 协议正文包含数据类型、目的、方式、存储和范围说明。

### 7.2 页面边界测试

通过加载页面模块并注入 `Page`、`wx`、授权模块与云调用 stub，覆盖：

- 未授权点击图片时选择和上传次数为 0。
- 未授权提交时业务云函数次数为 0。
- 授权有效时继续执行原有图片和保存流程。
- 勾选/取消分别调用授权写入/撤回接口。

### 7.3 回归验证

把新增测试纳入 `scripts/ci-build.sh`，完整运行现有语法、JSON、关系、删除、内容安全测试，避免隐私授权改动破坏已上线能力。

## 8. 发布约束

代码内协议不能替代微信公众平台后台的《小程序用户隐私保护指引》。发布人员必须确保后台指引与实际代码收集的数据一致，尤其是“选中的照片或视频信息”和云开发存储用途。
