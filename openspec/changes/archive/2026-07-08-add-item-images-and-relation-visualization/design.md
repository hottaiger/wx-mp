## Context

当前物品实体只有 `name / boughtAt / attrs / note / tags`，没有图片字段。用户希望在记“物”的时候能附一张图，这要求前端增加选图与上传流程，数据层增加图片 URL 或 fileID 存储。与此同时，详情页里的关联区还是一维列表，只展示关系标签，无法快速判断“这个人关联了哪些事、哪些物”，也看不到对端实体标题。

现有项目是微信小程序 + 云开发，图片上传最自然的方式是 `wx.chooseMedia` + `wx.cloud.uploadFile`，避免自建后端。关联展示已有双向数据，但前端缺少分组和摘要渲染层。

## Goals / Non-Goals

**Goals:**
- 让物品在创建和编辑时可上传 1 张主图
- 让物品列表卡片与详情页展示该主图
- 把详情页关联区按实体类型分组，特别优化“人”详情页
- 让关联卡片显示对端实体标题/名称与简短摘要

**Non-Goals:**
- 不做多图相册
- 不做图片裁剪、压缩策略后台优化
- 不做真正图谱可视化引擎（如力导图）
- 不做提醒链路的本轮补完

## Decisions

### 1. 物品图片模型

在 `items` 集合增加：

```js
coverImage: {
  fileID: string,
  tempUrl?: string
}
```

只保存 1 张主图。`fileID` 作为长期存储标识，前端读取时通过云文件临时 URL 或直接 fileID 渲染。

### 2. 图片上传流程

- `pages/capture` 和 `pages/detail` 中的物品表单新增“上传图片”入口
- 前端使用 `wx.chooseMedia({ count: 1, mediaType: ['image'] })`
- 选图后调用 `wx.cloud.uploadFile`
- 上传成功后把 `fileID` 写入 `item` 的 `coverImage.fileID`

理由：
- 复用微信云开发能力，避免额外服务
- 小程序天然支持云文件展示

### 3. 关联展示重构

详情页把 `relations` 分成三组：

- `events`
- `items`
- `persons`

每组为二级列表，组标题显示数量。组内卡片展示：

- 对端实体标题/名称
- 关系标签（如“拥有”“涉及”）
- 次级摘要（事件时间 / 物品标签 / 人物 traits）

对“人”详情页，优先渲染：
- 关联的事
- 关联的物
- 关联的人

### 4. 树状而非真图谱

本轮采用“分组树状列表”而不是图谱库：

```text
张三
├─ 关联的事（2）
│  ├─ 买鼠标
│  └─ 明天开会
├─ 关联的物（1）
│  └─ 鼠标
└─ 关联的人（1）
   └─ 李四
```

理由：
- 信息层级清晰
- 小程序实现成本低
- 不引入额外图形依赖

## Risks / Trade-offs

- **云存储临时 URL 失效** → 前端优先使用 fileID 渲染，必要时进入详情页再换取临时 URL
- **上传中断** → 表单保留本地状态，上传失败提示重试
- **关联组内需要额外查对端实体** → 详情页云函数聚合返回关系时一并补齐对端实体摘要，避免前端 N+1
- **树状展示不是完整图谱** → 本轮先解决可读性，后续再升级到真图谱

## Migration Plan

1. 更新 `item-entity` 和 `entity-relation` delta spec
2. 改造 `item` 云函数支持 `coverImage`
3. 在 `capture/detail/index` 实现图片上传与回显
4. 在 `detail/index` 重构关联分组 UI
5. 本地验证录入、查看、关联、删除

## Open Questions

- 是否需要物品无图占位图：默认需要
- 图片是否允许替换：允许，覆盖原主图
- 关联组默认展开还是折叠：默认展开
