# Brainstorm Summary

- Change: require-privacy-consent-before-capture
- Date: 2026-07-13

## 确认的技术方案

采用独立 `privacy-consent` 工具模块维护协议版本、结构化协议正文和本地授权记录；录入页使用未预选的协议勾选框与可滚动弹层。页面通过统一 `ensurePrivacyConsent()` 守卫同时保护物品图片上传和最终提交，授权前不发生云存储上传、订阅授权或业务云函数调用。用户已明确授权 agent 以后续可持续维护性为准完成设计决策。

## 关键取舍与风险

- 选择页面内弹层而非新增路由，保持现有页面路径稳定，同时让协议入口紧邻数据录入动作。
- 选择独立工具模块而非把协议正文和存储逻辑塞入页面，便于协议升级、单元测试和复用。
- 选择版本化本地授权记录，不引入服务端 schema；协议版本变化自动要求重新同意。
- 自定义协议只负责本次应用内告知与同意闭环，不能替代微信公众平台后台的用户隐私保护指引配置。

## 测试策略

- 使用 Node 内置测试覆盖协议版本匹配、无效记录、同意与撤回。
- 使用页面逻辑测试验证未授权时图片上传和业务云函数均不执行，授权后原流程保持可用。
- 运行 `bash scripts/ci-build.sh` 覆盖语法、JSON、既有关系与内容安全回归。

## Spec Patch

补充未授权物品图片上传场景，并明确统一授权守卫必须覆盖云存储上传和最终提交两个数据外传入口。
